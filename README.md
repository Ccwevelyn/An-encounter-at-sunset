## 匹配机制

所有匹配（随机 / 缘分）都遵循：**仅会匹配到「双向符合性别偏好」的人**。

- **希望匹配到的性别**：用户在档案中选择「男 / 女 / 不限」。系统要求：
  - 你选的偏好要包含对方性别；
  - 对方选的偏好也要包含你的性别；
  - 二者同时满足才会进入候选池。

### 随机匹配

- 双方都**开启随机模式**，且在 **5 分钟内**都点过「随机匹配」（视为同时段排队）。
- 在满足上述时间窗的前提下，先按**双向性别偏好**过滤，再在剩余用户中**随机**选一人匹配。
- 若该时间段内没有符合条件的用户，会提示「暂无同时段匹配的用户」。

### 缘分匹配

1. **性别**：同上，只保留双向符合性别偏好的用户。
2. **恋爱目的**  
   - 若你选「专注当下的快乐」：只与同样选「专注当下的快乐」的人匹配。  
   - 若你选「走向未来的婚姻」：只与同样倾向婚姻的人匹配，且需满足：  
     - **城市**：双方填写的城市至少有一个相同；  
     - **月花销档位**：按 A/B/C/D/E 档位，相邻档位可匹配（如 B 可匹配 A/B/C）。
3. **想恋爱指数**：两人指数差值 ≤ 2 才保留。
4. **恋爱经历（段数）**：两人经历档位差值 ≤ 2 才保留。
5. **优先级（在满足上述条件后）**  
   - 优先**同学位**；  
   - 再优先**最长恋爱时长**相近（档位差值 ≤ 2）；  
   - 再优先 **MBTI** 同类型或设定中的经典配对。  
6. 在最终候选池中**随机**选一人作为缘分匹配结果。

## 功能概览

- **登录 / 注册**：邮箱须为 `@mpu.edu.mo`，昵称，密码。
- **个人信息补充**：多步问卷（性别、希望匹配到的性别、学院、专业、生日、MBTI、感情经历、恋爱目的、城市、月花销、长大的城市、想恋爱指数等）。
- **两种匹配**：随机匹配（同时段 + 双向性别偏好）、缘分匹配（目的 / 城市 / 花销 / 恋爱指数 / 经历 / MBTI 等规则，见上）。
- **匹配结果**：查看对方档案，进入聊天。
- **个人页面**：展示已填资料与个人介绍，可编辑。
- **作者的话**：文档中的两段话与致谢。

### 数据库

当前使用 **SQLite**（`server/data.db`），数据保存在服务器本地。部署时请保证 `server` 目录可写，以便创建/写入 `data.db`。若以后要支持更大并发，可更换为 PostgreSQL/MySQL，并修改 `server/db.js` 与相关 SQL。

## 技术栈

- **前端**：React 18、React Router、Vite
- **后端**：Node.js、Express、better-sqlite3、bcryptjs、jsonwebtoken
- **部署**：Node 单进程即可；可加 Nginx + pm2 做生产部署
